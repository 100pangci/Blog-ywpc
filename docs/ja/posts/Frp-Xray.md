NAT 越えでは、FRP がよく使われる選択肢です。しかし、従来の FRP には2つの致命的な問題があります：
1. **トラフィックパターンが明確で、GFW に容易に遮断される**：TLS を使っても、各キャリアやファイアウォールは FRP の特徴を正確に識別でき、接続遮断や VPS IP のブラックリスト入りを引き起こします。
2. **ポートが完全に公開され、非常に危険**：どんなポートでも公開すれば、瞬時にスキャナーやスクリプトキディに狙われます。

これらの問題を解決するため、多くの人は「FRP をプロキシトンネルでラップする」という方法を考えます。これは簡単そうに聞こえますが、実際にはプロキシソフトウェアのルーティングルールによる多くの障壁に直面します。

今日は、非常に「欺瞞的」な高度アーキテクチャを紹介します：**「双方向偽装パブリック IP」の目くらましにより、Xray クライアントとサーバーの両方の制限を完璧に回避し、真の「ゼロポート露出」と「完全なトラフィック偽装」を実現します。**

---

## アーキテクチャの核心理念：Xray を欺く「舞台」

このソリューションの核心は、FRP トラフィックを Xray（または 3X-UI）のセキュアトンネル（VLESS-XTLS / Trojan など）に完全に隠すことです。FRP と Xray を完璧に連携させるため、2つの**完全に架空のパブリック IP** を導入します：
* **`22.22.22.22`**：**クライアント Xray** を欺き、FRP トラフィックを強制的にトンネルに通します。
* **`11.11.11.11`**：**サーバー Xray** を欺き、ローカルプライベート IP のアクセス制限（SSRF 対策）を回避し、最終的にカーネルが FRP サーバーにリダイレクトします。

アーキテクチャ全体では、Xray のプロキシポート（例：443）のみが公開され、FRP サーバー（frps）とすべてのトンネルサービスは VPS のローカルループバックインターフェースに「物理的に隠蔽」されます。

---

## ステップ1：サーバー側デプロイ — FRPS をローカルループバックに隠す

通常、`frps` を `127.0.0.1` で待機させることもできますが、後述の Xray のセキュリティ機構でブロックされます。そのため、Linux の機能を利用して、ローカルループバックインターフェース（`lo`）に仮想的なパブリック IP `11.11.11.11` を強制的にバインドします。

### 1. Systemd 起動スクリプトの修正
`/etc/systemd/system/frps.service` を編集し、サービス起動時と停止時にこの仮想 IP を動的に追加・削除します：

```ini
[Unit]
Description=Frp Server Service
After=network.target

[Service]
Type=simple
User=root

# 起動前に仮想 IP をローカル lo インターフェースにバインド
ExecStartPre=-/sbin/ip addr add 11.11.11.11/32 dev lo
ExecStart=/opt/frp/frps -c /opt/frp/frps.toml
# 停止後に仮想 IP を削除
ExecStopPost=-/sbin/ip addr del 11.11.11.11/32 dev lo

Restart=on-failure
RestartSec=5s
LimitNOFILE=1048576

[Install]
WantedBy=multi-user.target
```

### 2. FRPS の設定
FRP サーバーがこの仮想 IP のみで待機するようにします。外部の公網からは直接アクセスできません。

```toml
# /opt/frp/frps.toml
bindAddr = "11.11.11.11"
bindPort = 7000
proxyBindAddr = "11.11.11.11"

[auth]
method = "token"
token = "あなたの強力なパスワード"

[transport]
tls.force = false
tcpMux = true
heartbeatTimeout = 120
```

> **仕組みの解説（なぜ 11.11.11.11 を使うのか？）**
> サーバー側のプロキシソフトウェア（3X-UI など）には通常、SSRF 対策が組み込まれており、`127.0.0.1` などのローカルプライベート IP へのプロキシアクセスを**禁止**します。直接 127.0.0.1 にリダイレクトすると、Xray によってトラフィックが破棄されます。`11.11.11.11` という「偽装パブリック IP」を指定することで、Xray のセキュリティチェックはこれを正当な外部アドレスと見なして通過させます。Xray から出たトラフィックは、Linux カーネルがこの IP がローカル `lo` インターフェース上にあることを発見し、待機中の `frps` に正確に届けられます。

## ステップ2：クライアント側設定 — トラフィックを強制的にトンネルへ

クライアントマシン（自宅で Syncthing を実行している NAS や PC など）では、`frpc` がローカルの Xray プロキシ（例：ポート 1145 の SOCKS5 プロキシ）を経由するように設定します。

### FRPC 設定
```toml
# config/frpc.toml
serverAddr = "22.22.22.22"  # 重要：この IP は偽装です！
serverPort = 7000

[auth]
method = "token"
token = "あなたの強力なパスワード"

[transport]
proxyURL = "socks5://127.0.0.1:1145" # ローカルの Xray/v2ray クライアントを指定
tls.enable = false
tcpMux = true
heartbeatInterval = 30
heartbeatTimeout = 120

[[proxies]]
name = "syncthing"
type = "tcp"
localIP = "127.0.0.1"
localPort = 8384
remotePort = 8384
```

> **仕組みの解説（なぜ 22.22.22.22 を使うのか？）**
> ほとんどの Xray クライアントには優先度の高いルーティングルール `geoip:private -> direct`（プライベート IP は直接接続）があります。ここに `127.0.0.1` や `192.168.x.x` を指定すると、Xray クライアントはこれを内部アドレスと判断し、プロキシトンネルをスキップしてしまい、FRP は外部ネットワークに接続できなくなります。偽装されたパブリック IP `22.22.22.22` を指定することで、Xray クライアントをうまく欺き、トラフィックを暗号化して VPS に送信させることができます。

## ステップ3：サーバー側ゲートウェイルーティング — 3X-UI による精密リダイレクト

外部のデバイス（プロキシに接続しているスマートフォンなど）から自宅の Syncthing（ポート 8384）にアクセスする場合、リクエストは `22.22.22.22:8384` に送られ、VPS 上の 3X-UI に到達します。

ここで 3X-UI にルーティングルールを設定し、このトラフィックを隠蔽された FRPS に「誘導」する必要があります。

**1. ルーティングルールの作成：**
* ネットワーク：`tcp`（または any）
* 宛先 IP：`22.22.22.22`
* 宛先ポート：`8384`
* アウトバウンドタグ：`to-8384`

**2. 対応するアウトバウンドルールの作成：**
* プロトコル：`freedom`（直接接続）
* タグ：`to-8384`
* **リダイレクト**：`11.11.11.11:8384`

※複数の FRP マッピングポート（7000、8384 など）がある場合は、必要に応じて複数の Redirect アウトバウンドとルーティングルールを作成するか、正規表現でマッチングさせます。

## 完全なデータフロー図

これで、この精巧な「欺瞞ゲーム」が完成しました。完全なデータフローは以下の通りです：

```text
[クライアント（スマートフォン等）] -> 22.22.22.22:8384 にリクエスト
       │
   (暗号化トンネル)
       ▼
[VPS: Xray/3X-UI サーバー] 
       │ 
       ├─ (ルートマッチ 22.22.22.22:8384 -> アウトバウンド to-8384)
       │ 
       └─ (リダイレクト実行：11.11.11.11:8384)
       │
[VPS: Linux カーネルが処理] -> 11.11.11.11 がローカル lo にあることを検出
       ▼
[VPS: frps サーバー (11.11.11.11 で待機)]
       │
   (FRP プロトコルトンネル - Xray プロキシトンネル内部で動作)
       ▼
[ローカル: frpc クライアント] 
       │
       ▼
[ローカル: Syncthing サービス (127.0.0.1:8384)]
```
